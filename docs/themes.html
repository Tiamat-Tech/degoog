Where themes live

Themes live in data/themes/ (or DEGOOG_THEMES_DIR). Each theme is a folder named by its id (e.g. zen, catpuccin), containing at least a theme.json manifest.

Theming tiers

There are three ways to theme degoog, from simplest to most powerful. Pick the tier that matches what you need.

Tier What you provide What you control Example
CSS only theme.json + stylesheet Colours, fonts, spacing via CSS variables Catppuccin, Pokemon, Sakura
CSS + templates theme.json + stylesheet + template files Everything above, plus the HTML structure of individual results, image cards, video cards, and the at-a-glance box —
Full HTML override theme.json + stylesheet + custom search.html / index.html Total control over the page. Templates can be placed inline inside the custom HTML. Zen

Theme manifest (theme.json)

Required:

  • name — Display name in Settings → Themes.

Optional:

  • author, description, version — Shown on the theme card.
  • css — Path to the stylesheet relative to the theme folder (e.g. style.css or style.scss). SCSS is compiled at load. When the theme is active this file is served at /theme/style.css.
  • templates — Object mapping template ids to file paths relative to the theme folder. These template files are injected into the default page as <template> elements, so you do not need to provide a custom search.html. See Templates below.
  • html — Object with keys index, search. Value is the filename in the theme folder (e.g. "index": "index.html"). When the theme is active, that file is served instead of the built-in page. Custom HTML can use page placeholders: __THEME_CSS__, __THEME_ATTRS__, __PLUGIN_ASSETS__, __APP_VERSION__, __THEME_TEMPLATES__.
  • settingsSchema — Same shape as plugin SettingFields. If present, a Configure button appears on the theme card; values are stored in data/plugin-settings.json under theme-<theme-id>. Settings can be exposed as data-* attributes on <html> via dataAttrsFromSettings.

Minimal CSS-only example

{
  "name": "My Theme",
  "css": "style.css"
}

CSS + templates example

{
  "name": "My Theme",
  "css": "style.css",
  "templates": {
    "result": "templates/result.html",
    "image-card": "templates/image-card.html"
  }
}

Full HTML override example

{
  "name": "My Theme",
  "css": "style.css",
  "html": {
    "index": "index.html",
    "search": "search.html"
  }
}

Templates

Templates let you control the HTML markup of search results, media cards, and other dynamic elements without replacing the entire page. The JavaScript fills in data using placeholders — you just write HTML.

There are two ways to provide templates:

  1. Standalone files (recommended) — list them in the templates object in theme.json. The server injects them into the default page automatically. No search.html override needed.
  2. Inline in custom HTML — if your theme overrides search.html, place <template> elements directly in your HTML. Use __THEME_TEMPLATES__ as a placeholder or put them anywhere before </body>.

Template IDs

Each template is identified by its element id. The templates object in theme.json maps a short key to a file path; the key is automatically prefixed with degoog- to form the id. For inline templates, use the full id directly.

Manifest key Element id Used for
result degoog-result Web / news search result item
image-card degoog-image-card Image grid card
video-card degoog-video-card Video grid card
at-a-glance degoog-at-a-glance At-a-glance box

Placeholder syntax

Inside a <template> element, use double curly braces for dynamic values. All values are HTML-escaped automatically.

Syntax Description
{{ name }} Output a value (escaped)
{{#if name}} … {{/if name}} Conditional block — rendered only when the value is truthy (non-empty string, non-zero, non-empty array)
{{#each name}} … {{/each name}} Loop over an array. Inside the loop, {{ . }} is the current item and {{ @index }} is the index.

Important: because templates live inside HTML <template> elements, the browser parses them as HTML. This means:

  • {{#if}} blocks must contain balanced HTML — every opening tag inside the block must have its closing tag inside the same block.
  • Placeholders work inside text content and inside quoted attribute values (href="{{ url }}"), but not between attributes or as tag names.

Example: custom result template

Create templates/result.html in your theme folder: 

<div class="my-result">
  <a href="{{ url }}" target="{{ link_target }}">
    <strong>{{ title }}</strong>
  </a>
  <p>{{ snippet }}</p>
  <small>{{ cite_url }}</small>
  {{#if thumbnail_url}}
  <img src="{{ thumbnail_url }}" loading="lazy">
  {{/if thumbnail_url}}
  <div>
    {{#each sources}}
    <span class="tag">{{ . }}</span>
    {{/each sources}}
  </div>
</div>

Then in theme.json:

{
  "name": "My Theme",
  "css": "style.css",
  "templates": {
    "result": "templates/result.html"
  }
}

Only the result template is overridden — image cards, video cards, and the at-a-glance box keep their defaults.

Inline templates (full HTML override)

If your theme provides a custom search.html, you can place <template> elements anywhere in the page. Use the __THEME_TEMPLATES__ placeholder to inject any standalone template files from other themes that extend yours, or omit it if you want full control. 

<!-- inside your search.html, before </body> -->
<template id="degoog-result">
  <div class="my-result">
    <a href="{{ url }}">{{ title }}</a>
    <p>{{ snippet }}</p>
  </div>
</template>

Setup

Create data/themes/ (or set DEGOOG_THEMES_DIR). Each theme is a subfolder, e.g. data/themes/my-theme/, with:

  • theme.json (required)
  • style.css or style.scss (optional) — Use the app's CSS variables so light/dark mode work. See Styling for the full list.
  • templates/ (optional) — Template files referenced by theme.json.
  • index.html, search.html (optional) — Only needed for full HTML override; listed under html in theme.json.

The theme id is the folder name.

Applying a theme

Settings → Themes → choose a theme → Apply. The active theme id is stored in data/plugin-settings.json as theme.active.

Migrating existing themes

Existing themes that use only CSS or full HTML overrides continue to work with no changes. To take advantage of templates:

  1. If your theme overrides search.html only to restyle results, consider switching to CSS + template files instead. This makes your theme resilient to upstream page layout changes.
  2. Extract the result markup from your custom search.html into template files, replacing hardcoded data with placeholders.
  3. Add a templates object to your theme.json and remove the html entry if the page override is no longer needed.

Examples from the official store

  • Catppuccin, Pokemon, Sakura — CSS-only themes with variant settings. No HTML or template overrides.
  • Zen — Full HTML override with custom index.html and search.html.
